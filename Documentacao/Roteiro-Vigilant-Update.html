Publicar uma Nova VersaoProcedimento para o desenvolvedor

1.1 Estrutura de Pastas do Repositorio

1.2 Como o post-install.sh Funciona

O post-install.sh copia os arquivos do pacote para os destinos corretos no sensor. E chamado automaticamente pelo vigilant-updater.sh apos download e verificacao — nunca precisa ser executado manualmente.

O que colocar em cada lugar

O que voce quer fazerOnde colocar no repositorio
Adicionar/atualizar script de operacaosensor/scripts/
Atualizar regras do Snortsensor/configs/snort/rules/
Atualizar config do Cowriesensor/configs/cowrie/
Remover arquivo obsoleto do sensorsensor/removals.txt (caminho absoluto por linha)
Executar comandos especificos nesta versaosensor/custom-deploy.sh
⚠
Sobrescrita sem confirmacao O post-install.sh sobrescreve arquivos existentes sem pedir confirmacao. Se um subdiretorio de configs/ estiver vazio no pacote, ele e ignorado — nao apaga nada no sensor.

1.3 Adicionar ou Modificar um Arquivo

Exemplo: adicionar meu-script.sh no sensor.

# Na raiz do repositorio
cd /caminho/para/sensores-updates

# Copie ou crie o script na pasta correta
cp /origem/meu-script.sh sensor/scripts/meu-script.sh

# Verifique o que mudou
git status
git diff --stat
ⓘ
Regra de mapeamento Tudo dentro de sensor/scripts/ vai parar em /vigilant/scripts/ no sensor. Se for config de servico, use sensor/configs/<servico>/.

1.4 Atualizar o CHANGELOG

Edite sensor/CHANGELOG.md antes de publicar o tag:

## vX.Y.Z (YYYY-MM-DD)
- Adicionado meu-script.sh para [descricao]

1.5 Fazer Commit e Publicar o Tag

# Adicionar os arquivos modificados
git add sensor/scripts/meu-script.sh sensor/CHANGELOG.md

# Commit
git commit -m "feat: adicionar meu-script.sh — descricao breve"

# Tag — escolha o numero correto:
# Patch (bug/adicao pequena):  v2.0.3 → v2.0.4
# Minor (nova funcionalidade): v2.0.3 → v2.1.0
# Major (breaking change):     v2.0.3 → v3.0.0
git tag v2.0.4

# Push do commit e do tag (dispara CI automaticamente)
git push origin main
git push origin v2.0.4

1.6 Acompanhar o CI no GitHub

  • 1Acesse github.com/Vigilant-devs/sensores-updates/actions
  • 2Aguarde o workflow Build, Sign & Publish Release — leva ~1-2 minutos
  • 3Check verde: nova Release criada e manifest.json atualizado para a nova versao
⚠
Nunca edite o manifest.json manualmente Ele e atualizado exclusivamente pelo CI. Edicoes manuais serao sobrescritas e podem quebrar o fluxo.

2.2 Disparar a Atualizacao Manualmente

# Forcar execucao imediata
systemctl start vigilant-updater.service

# Acompanhar em tempo real
journalctl -u vigilant-updater.service -f

2.3 Verificar se a Atualizacao Foi Aplicada

# Versao deve estar atualizada (ex: 2.0.4)
cat /vigilant/scripts/vigilantsensor/updater/VERSION

# Verificar se o novo script chegou
ls -la /vigilant/scripts/meu-script.sh

# Log completo do evento
tail -50 /vigilant/scripts/vigilantsensor/logs/vigilant-update.log

Formato do Log (JSON)

{
  "timestamp": "...",
  "sensor":    "sensor-1",
  "client":    "20004",
  "event":     "update_success",
  "version_from": "2.0.3",
  "version_to":   "2.0.4",
  "details":   "..."
}

Eventos Possiveis

EventoSignificado
no_updateSensor ja esta na versao mais recente — nenhuma acao
update_successAtualizacao aplicada com sucesso
rollbackFalha detectada — versao anterior restaurada automaticamente
verify_failedChecksum ou assinatura GPG invalida — pacote rejeitado

2.4 Confirmar no Grafana

Acesse https://172.16.162.189 — login admin / (senha da instalacao)

Painel Vigilant Sensor Updates:

  • Total de Updates — deve incrementar +1
  • Log de Eventos — evento update_success com version_to: 2.0.4
  • Versao Atual por Sensor — coluna "Versao" deve mostrar 2.0.4
3

Cenarios de TesteCasos de uso e verificacoes esperadas

A

Script Novo Adicionado

Verificar que novo script chega ao sensor com permissao correta.

  1. Adicione sensor/scripts/teste-novo.sh com echo "ok"
  2. Publique a versao (Parte 1)
  3. Dispare o updater (Parte 2)
  4. Verifique: ls -la /vigilant/scripts/teste-novo.sh — deve ter +x
B

Script Existente Modificado

Verificar que modificacao sobrescreve o arquivo no sensor.

  1. Edite um script em sensor/scripts/
  2. Publique nova versao
  3. Anote conteudo atual: head -5 /vigilant/scripts/firewall.sh
  4. Dispare o updater
  5. Confirme que o arquivo foi atualizado
C

Sensor Ja Atualizado

Verificar que o updater nao faz nada desnecessario.

  1. Sensor ja na versao mais recente — dispare o updater
  2. Log deve mostrar no_update
  3. Versao nao muda
D

Atualizacao Agendada (Timer)

Verificar que o timer automatico esta funcionando.

  1. systemctl status vigilant-updater.timer
  2. Se inactive: systemctl enable --now vigilant-updater.timer
  3. Para testes rapidos ative vigilant-updater-test.timer
  4. Confirme: systemctl list-timers | grep vigilant

Configurar Timer de Laboratorio Diretamente no Sensor

ssh -p 12222 root@172.16.162.197

# Criar timer de teste (10 minutos)
cat > /etc/systemd/system/vigilant-updater-test.timer << 'EOF'
[Unit]
Description=Vigilant Sensor Updater - Test Mode (every 10 minutes)

[Timer]
Unit=vigilant-updater.service
OnBootSec=1min
OnUnitActiveSec=10min
Persistent=true

[Install]
WantedBy=timers.target
EOF

# Recarregar e ativar o timer de teste
systemctl daemon-reload
systemctl disable vigilant-updater.timer
systemctl enable  --now vigilant-updater-test.timer

# Confirmar agendamento
systemctl list-timers | grep vigilant

Saida esperada:

NEXT                         LEFT      LAST                         PASSED
Thu 2026-03-05 XX:XX:00 BRT  9min 47s  Thu 2026-03-05 XX:XX:00 BRT  13s ago
vigilant-updater-test.timer
✓
Restaurar timer de producao apos testes systemctl disable --now vigilant-updater-test.timer
systemctl enable --now vigilant-updater.timer

TroubleshootingDiagnostico e resolucao de problemas comuns

⚠  Updater nao executa
# Ver erro do servico
systemctl status vigilant-updater.service
journalctl -u vigilant-updater.service --no-pager -n 50

# Verificar conectividade com GitHub
curl -s https://raw.githubusercontent.com/Vigilant-devs/sensores-updates/main/manifest.json | jq .
⚠  no_update quando deveria atualizar
# Versao local
cat /vigilant/scripts/vigilantsensor/updater/VERSION

# Versao no manifest (o que o sensor ve)
curl -s https://raw.githubusercontent.com/Vigilant-devs/sensores-updates/main/manifest.json | jq .version

# Se iguais: CI ainda nao terminou — aguarde e tente novamente
# Se diferentes: inspecionar log completo para ver o motivo
⚠  Evento verify_failed

O pacote foi baixado mas SHA256 ou assinatura GPG nao batem. Causas possiveis:

  • CI falhou no step de assinatura — verificar no GitHub Actions
  • Chave publica no sensor esta desatualizada
# Ver chave publica no sensor
ls -la /vigilant/scripts/vigilantsensor/updater/vigilant.pub.gpg
⚠  Logs nao aparecem no Grafana
# No sensor: verificar se rsyslog esta encaminhando
systemctl status rsyslog
cat /etc/rsyslog.d/50-vigilant-updater.conf

# No servidor de logs: verificar recebimento
tail -20 /var/log/vigilant/sensor-updates.log
systemctl status promtail
